日志查看

成功还原后，就可以开始看日志了

分析死锁，着重看LATEST DETECTED DEADLOCK最近一次死锁日志

根据上图可知事务2：

  • 造成死锁的原因是正在等待插入意向锁


根据上图可知事务1：

  • 正在等待插入意向锁
  • 死锁解开后，根据0: len 8; hex 73757072656d756d; asc supremum;;可知事务1持有gap lock，且锁住的范围为（最大记录值，正无穷），即（10002，+∞）。


根据上图可吃，最终回滚了事务1

通过日志能够知道是因为两个事务在互相等待意向插入锁导致的死锁，但造成这种现象还需进一步分析：

  • 事务1执行完update语句后，持有（10002，+∞）的gap lock，更新成功
  • 事务2执行完update语句后，持有（10002，+∞）的gap lock，更新成功。PS：gap lock之间是兼容的
  • 事务1执行insert语句时，需要获取（10002，+∞）的插入意向锁，但事务2持有着，阻塞等待
  • 事务2执行insert语句时，需要获取（10002，+∞）的插入意向锁，但事务1持有着，阻塞等待
  • 由于两个事务互相持有一个相同范围的gap lock，有形成互相等待情况，所以死锁发生
  • innodb自动解除死锁状态，选择事务1进行回滚，故事务2能够获取（10002，+∞）的插入意向锁，插入成功。

暖心送上各种锁之间的兼容情况：
| 兼容性 | Gap | Insert Intention | Record | Next-key |
| —————- | — | —————- | —— | ——– |
| Gap | 兼容 | 兼容 | 兼容 | 兼容 |
| Insert Intention | 冲突 | 兼容 | 兼容 | 冲突 |
| Record | 兼容 | 兼容 | 冲突 | 冲突 |
| Next-key | 兼容 | 兼容 | 冲突 | 冲突 |

线上case解决方案

解决方案其实异常的简单：将update历史数据的逻辑进行优化，不能无脑更新，在更新前需要先进行select，如果存在历史数据时，则需要update操作，避免了互相加上gap lock的情况。

总结

最后，遇到死锁问题，我们应该怎么分析呢？

  • 模拟死锁场景
  • show engine innodb status;查看死锁日志
  • 找出死锁SQL
  • SQL加锁分析，这个可以去官网看哈 https://dev.mysql.com/doc/refman/5.7/en/innodb-locks-set.html
  • 分析死锁日志（持有什么锁，等待什么锁）
  • 熟悉锁模式兼容矩阵，InnoDB存储引擎中锁的兼容性矩阵。
